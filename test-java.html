Testing trong Java – Các Công Cụ và Phương Pháp

Testing không chỉ “bắt bug” mà còn thiết kế API rõ ràng, giữ chất lượng khi refactor và tăng tự tin khi release. Dưới đây là loại kiểm thử, công cụ phổ biến, phương pháp TDD/BDD, thực hành tốt, kèm ví dụ ngắn.

Nội dung
  • 1) Các loại kiểm thử
  • 2) Công cụ phổ biến
  • 3) Phương pháp kiểm thử
  • 4) Best Practices
  • 5) Ví dụ nhanh (JUnit + Mockito)

1) Các Loại Kiểm Thử (Testing Types)

  • Unit Testing: kiểm thử đơn vị, nhắm vào hàm/lớp nhỏ, nhanh và độc lập.
  • Integration Testing: kiểm thử tích hợp giữa các thành phần (DB, API, message queue…).
  • System Testing: kiểm thử toàn hệ thống theo luồng nghiệp vụ.
  • Acceptance Testing: kiểm thử chấp nhận theo tiêu chí người dùng/doanh nghiệp.

2) Các Công Cụ Kiểm Thử Phổ Biến trong Java

  • JUnit: tiêu chuẩn de-facto cho unit test (JUnit 5/Jupiter).
  • Mockito: giả lập (mock) phụ thuộc để cô lập đơn vị cần test.
  • TestNG: linh hoạt nhóm bài test, chạy song song, cấu hình phong phú.
  • Cucumber: BDD, mô tả kịch bản bằng ngôn ngữ dễ đọc (Gherkin).

3) Phương Pháp Kiểm Thử

  • TDD (Test-Driven Development): Red → Green → Refactor. Viết test thất bại trước (Red), viết mã cho pass (Green), rồi refactor có test bảo vệ.
  • BDD (Behavior-Driven Development): mô tả hành vi mong đợi bằng kịch bản Given–When–Then, tăng tính thống nhất giữa kỹ thuật và nghiệp vụ.

4) Các Best Practices trong Kiểm Thử Java

  • Test nhanh, độc lập, không phụ thuộc thứ tự chạy.
  • Đặt tên test như một mệnh đề hành vi: shouldCalculateTotal_whenCartHasItems.
  • Dùng mock cho phụ thuộc bên ngoài; tích hợp thật ở integration test.
  • Fixture gọn gàng (builder/test data factory), tránh lặp setup rườm rà.
  • Chạy test trong CI, báo cáo coverage để theo dõi xu hướng (không chạy theo % mù quáng).

5) Ví dụ nhanh: JUnit 5 + Mockito

JUnit 5: test đơn giản cho lớp Calculator.

import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class Calculator {
    int add(int a, int b) { return a + b; }
}

class CalculatorTest {
    @Test
    void add_shouldReturnSum() {
        Calculator c = new Calculator();
        assertEquals(7, c.add(3, 4));
    }
}

Mockito: mock phụ thuộc để cô lập UserService.

import static org.mockito.Mockito.*;
import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class User { private final String name; User(String n){ this.name=n; } String getName(){ return name; } }
interface UserRepo { User findById(long id); }

class UserService {
    private final UserRepo repo;
    UserService(UserRepo repo){ this.repo = repo; }
    String getName(long id){ User u = repo.findById(id); return u == null ? null : u.getName(); }
}

class UserServiceTest {
    @Test
    void getName_shouldReturnFromRepo() {
        UserRepo repo = mock(UserRepo.class);
        when(repo.findById(1L)).thenReturn(new User("Ann"));

        UserService svc = new UserService(repo);
        assertEquals("Ann", svc.getName(1L));
        verify(repo).findById(1L);
    }
}

Gợi ý: Với Spring, dùng @WebMvcTest/@DataJpaTest để kiểm thử lát cắt; với DB thật, cân nhắc Testcontainers.
